# Module: Runner

> Generated by Cicadas — 2026-03-24.

## Purpose

The Runner module is the CLI entry point and pipeline orchestrator. It parses operator commands, validates step combinations, sequences the pipeline steps, and manages run lifecycle (start, stop, run ID tracking).

## Key Components

### `runner.py` — CLI and orchestration

**Entry point**: `main()` — parses CLI args with `click`, dispatches to sub-commands.

**`Steps` enum**:
```python
HARVEST        = "harvest"        # scrape + clean (combined)
HARVEST_SCRAPE = "harvest_scrape" # scraping only
HARVEST_CLEAN  = "harvest_clean"  # cleaning only
REHARVEST      = "re-harvest"     # re-clean existing HTML
EXTRACT        = "extract"        # headline extraction + article collection
INTERPRET      = "interpret"      # single-run LLM analysis
INTERPRET_WEEKLY = "interpret_weekly"  # weekly LLM analysis (hybrid dates)
SUMMARIZE_DAILY  = "summarize_daily"   # daily prose summary
FORMAT         = "format"         # HTML generation
DEPLOY         = "deploy"         # SFTP upload
```

**Key async functions**:

| Function | Description |
|----------|-------------|
| `run(steps, sites, **kwargs) -> dict` | Main pipeline orchestrator. Returns `{"run_id", "status", "completed_steps", "error"}` |
| `scrape(sites) -> str` | Scraping phase only, returns job_dir |
| `clean(job_dir, sites)` | Cleaning phase only |
| `extract(job_dir)` | Creates `ContextExtractor`, runs extraction with 10s delay between sites |
| `interpret(job_dir, sites)` | Creates `LLMWebsiteInterpreter`, runs `interpret_files()` |
| `interpret_weekly(use_rolling_daily, specific_weeks, sites)` | Weekly interpretation with hybrid date logic |
| `format_output(force_full, sites, job_dir)` | HTML generation with cursor support |
| `deploy_output(force_full, job_dir, sites)` | SFTP upload with cursor support |
| `summarize_all(force)` | Runs `DailySummarizer` across all job dirs |
| `reinterpret_weeks_from_date(start_date, overwrite)` | Batch reinterpretation from a given date |

**CLI commands**:
```
run -s STEPS [OPTIONS]
    --job-dir, -j DIR           Process specific job directory
    --start-date DATE           Date range start
    --end-date DATE             Date range end
    --sites SITE...             Override default sites
    --playwright-mode MODE      local|cloud
    --force-full-format         Ignore format cursor
    --force-full-deploy         Ignore deploy cursor
    --rewind-days N             Rewind cursors N days before running
    --run-id ID                 Assign custom run ID

summarize [-f|--force]
reinterpret-weeks -d DATE [--no-overwrite]
stop
reset-cursor [--format] [--deploy] [--all]
audit [--start-date DATE] [--end-date DATE] [--no-report]
```

**Step validation** (`validate_step_combinations(steps)`):
- ❌ `harvest + harvest_scrape` — conflicting (harvest includes scraping)
- ❌ `harvest + harvest_clean` — conflicting (harvest includes cleaning)
- ❌ `harvest_scrape + harvest_clean` — redundant (use `harvest` instead)

### `RunState` (`common.py`)
Static class for global run lifecycle management.

- `RunState.reset(run_id: Optional[str])` — starts a new run, resets stop flag
- `RunState.request_stop()` — sets stop flag; pipeline checks this between steps
- `RunState.stop_requested() -> bool` — checked between each step in the pipeline
- `RunState.get_run_id() -> str` — returns current run UUID

### `cloud_entrypoint.py` — Flask HTTP API
Wraps the pipeline in an HTTP server for cloud scheduling.

- Flask app on port 8080
- Spawns pipeline in background threads via `run_task_async()`
- Maintains `active_runs: dict[str, RunTaskState]` for status tracking
- Exposes: `GET /`, `GET /health`, `POST /run`, `POST /weekly`, `POST /summarize`, `POST /stop/{run_id}`, `GET /status`
- Used by GCP Cloud Scheduler cron jobs → triggers daily pipeline execution

## Interfaces

**Exports**: CLI commands and HTTP endpoints (external interfaces, not imported by other modules)

**Dependencies**:
- All other modules — runner orchestrates the full pipeline
- `common.RunState` — run lifecycle management
- `storage.shared_storage` — storage instance initialization

## Key Decisions

- **Steps are individually addressable**: The `-s` flag accepts any subset of steps in any valid order. This is critical for reprocessing (e.g., re-running `format deploy` without re-scraping).
- **`extract` has a 10-second delay between sites**: Rate limiting for the LLM API calls in `ContextExtractor`. Hardcoded in `extract()` function.
- **`run()` returns a dict with `run_id`**: This enables the Flask API to track and query run status asynchronously after the pipeline is submitted.
- **`RunState` is static (not instance-based)**: There is only ever one active pipeline run per process. Multiple concurrent runs are not supported.
- **Cloud VM, not Cloud Scheduler directly**: The cron job triggers `POST /run` via HTTP rather than running a Python process directly. This allows the same container to serve the API while running the pipeline.
