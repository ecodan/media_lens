# Module: Collection

> Generated by Cicadas — 2026-03-24.

## Purpose

The Collection module handles the **Harvest phase**: acquiring raw web content (scraping) and normalizing it for analysis (cleaning). It produces cleaned HTML files per site per run, written to the job directory.

## Key Components

### `WebpageScraper` (`collection/scraper.py`)
Fetches a URL using Playwright with stealth mode to bypass bot detection.

- **Static method**: `get_page_content(url: str, browser_type: BrowserType) -> Optional[str]`
- **BrowserType enum**: `DESKTOP` (1920×1080, Windows UA) or `MOBILE` (375×812, iPhone 12 UA). Mobile preferred — produces cleaner layouts.
- **Playwright mode** (via `PLAYWRIGHT_MODE` env): `local` (minimal flags for macOS) or `cloud` (no-sandbox, single-process for containers)
- **Stealth**: `playwright-stealth` applied to avoid bot fingerprinting
- Returns raw HTML string or `None` on timeout/error

### `WebpageCleaner` (`collection/cleaner.py`)
Applies site-specific HTML cleaning to strip boilerplate, ads, navigation, and non-article content.

- **Main method**: `clean_html(html: str, site: str) -> str`
- Strips `<script>` and `<style>` tags first
- Applies site-specific `SiteSpecificCleaner` pattern matching
- **Truncates to 100KB** before returning — prevents Vertex AI quota exhaustion
- Returns prettified HTML with semantic structure preserved

### `SiteSpecificCleaner` hierarchy (`collection/cleaning.py`)
Strategy pattern for per-outlet cleaning rules.

- **Abstract**: `SiteSpecificCleaner.clean_page(page: BeautifulSoup) -> BeautifulSoup`
- **`PatternBasedCleaner`**: Constructor takes `patterns: list[str]` (CSS selectors). Keeps matching elements and their ancestors; decomposes everything else.
- **Concrete cleaners**: `CNNCleaner`, `BBCCleaner`, `FoxNewsCleaner` — each with hard-coded CSS selector patterns
- **`CleanerConfig.get_cleaner(site: str) -> SiteSpecificCleaner`**: Factory method

### `Harvester` (`collection/harvester.py`)
Orchestrates the two-phase collection workflow.

- `harvest(sites: list[str], browser_type: BrowserType) -> str` — sequential scrape → clean, returns job_dir
- `scrape_sites(sites: list[str], browser_type: BrowserType) -> str` — creates timestamped job dir, saves `{site}.html` per site
- `clean_sites(job_dir: str, sites: list[str]) -> None` — reads `{site}.html`, applies `WebpageCleaner`, saves `{site}-clean.html`
- `re_harvest(job_dir: Path, sites: list[str]) -> None` — re-cleans existing HTML without re-downloading (for updated cleaner patterns)

## Interfaces

**Exports**:
- `Harvester` — entry point for the collection step in `runner.py`
- `WebpageScraper` — also used directly by `ArticleCollector` (extraction layer) to bypass Cloudflare blocks on article URLs
- `WebpageCleaner` — used by `Harvester.clean_sites()`

**Dependencies**:
- `storage_adapter.py` (via `storage.shared_storage`) — all file writes go through the storage abstraction
- `common.py` — logger, `SITES_DEFAULT`

## Key Decisions

- **`WebpageScraper` is used by both Collection and Extraction layers**: `ArticleCollector` imports `WebpageScraper` directly to fetch article URLs that trafilatura can't access due to Cloudflare 403 blocks. This is intentional cross-layer dependency.
- **100KB truncation happens in WebpageCleaner**: This is the last step in cleaning, ensuring all downstream consumers (LLM calls) receive capped input regardless of scraper output size.
- **Mobile viewport preferred**: `BrowserType.MOBILE` tends to produce less cluttered HTML with cleaner article structure. Check `Harvester` to see which mode is passed from `runner.py`.
- **Site-specific cleaners are CSS-selector-based, not DOM-heuristic**: This means adding a new site requires writing a concrete `SiteSpecificCleaner` subclass with known CSS selectors for that outlet's article containers.
