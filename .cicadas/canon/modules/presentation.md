# Module: Presentation

> Generated by Cicadas — 2026-03-24.

## Purpose

The Presentation module handles the final two pipeline steps: **format** (generating static HTML from JSON data) and **deploy** (uploading HTML to the remote SFTP web server). Both steps use cursor-based incremental processing to skip unchanged content.

## Key Components

### `HtmlFormatter` (`presentation/html_formatter.py`)
Generates static HTML reports from interpretation and summary JSON data.

**Main entry point**:
- `generate_html_from_path(sites, template_dir_path, force_full, job_dir)`:
  1. Reads format cursor (or skips if `force_full=True`)
  2. Finds all job dirs newer than cursor
  3. Groups by ISO week
  4. For each affected week: generates weekly report + individual article pages
  5. Regenerates landing page (always reflects rolling 7-day)
  6. Updates format cursor

**HTML outputs** (written to `staging/`):
- `staging/medialens.html` — Landing page, rolling 7-day view
- `staging/medialens-{YYYY-WNN}.html` — Weekly archive page per ISO week
- `staging/articles/{site}/{year}/{month}/{day}/{time}/{filename}.html` — Article reader pages

**Template rendering**:
- `generate_html_with_template(template_dir_path, template_name, content)` — Jinja2 rendering, returns HTML string
- Templates: `index_template.j2`, `weekly_template.j2`, `article_template.j2` (in `config/templates/`)

**Article page generation**:
- `generate_article_page(story, site, job_dir_path, template_dir_path) -> Optional[str]`
  - Reads article JSON from storage
  - Renders with `article_template.j2`
  - Returns relative URL path for linking in weekly/index reports
  - Returns `None` if article data unavailable

**Cursor management**:
- `get_format_cursor() -> Optional[datetime]` — reads `format_cursor.txt`
- `update_format_cursor(timestamp: datetime)` — writes after successful run
- `reset_format_cursor()` — deletes cursor, forces full regeneration
- `rewind_format_cursor(days: int)` — rewinds cursor N days back

### `Deployer` (`presentation/deployer.py`)
Uploads generated HTML from `staging/` to remote SFTP server via paramiko.

**SFTP configuration env vars**: `FTP_HOST`, `FTP_USER`, `FTP_PASSWORD`, `FTP_REMOTE_PATH`

**Key functions**:
- `upload_html_content_from_storage(storage_path: str) -> bool` — reads HTML from storage, uploads to SFTP with automatic nested directory creation
- `get_files_to_deploy(cursor: Optional[datetime]) -> List[str]` — lists `staging/**/*.html`, filtered by cursor timestamp

**Cursor management** (mirrors format cursor):
- `get_deploy_cursor()`, `update_deploy_cursor()`, `reset_deploy_cursor()`, `rewind_deploy_cursor(days)`

## Interfaces

**Exports** (consumed by `runner.py`):
- `generate_html_from_path()` — `format` step
- `Deployer` (or its top-level functions) — `deploy` step

**Dependencies**:
- `storage.shared_storage` — all file reads/writes including staging/ directory
- `common.py` — logger, week utilities (`get_week_key`, `get_week_display`)
- `job_dir.JobDir` — job directory discovery and grouping by week

## Key Decisions

- **Cursors prevent redundant work in production**: The format and deploy steps are run daily. Without cursors, all historical weeks would be regenerated and re-uploaded on every run. This is essential for cost and time efficiency.
- **Landing page is always regenerated**: Even with cursor-based incremental processing, `medialens.html` is always regenerated because it shows a rolling 7-day window that changes every day regardless of new job dirs.
- **GCS rate limiting mitigation**: `generate_article_page` previously called `storage.create_directory()` before writes; this was removed to avoid GCS 429 rate limiting errors (GCS doesn't require explicit directory creation).
- **SFTP auto-directory creation**: `upload_html_content_from_storage` creates all necessary remote directories before uploading. The deployer handles arbitrarily deep `articles/` directory trees.
- **Staging/ is the buffer**: All format output goes to `staging/`. The deployer reads from `staging/`. This decouples generation from deployment and enables re-deployment without re-formatting.
